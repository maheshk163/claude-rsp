Intro & Helios orientation
~45 sec
What to show on screen
  • Open Cohesity Helios at the main dashboard landing page
  • Show the global cluster/node overview — all systems green
  • Hover over the Oracle source count to confirm it's registered
Script

"This is Cohesity Helios — our unified control plane for all Oracle database backups. What you're looking at is a single dashboard that replaces the scattered cron emails and manual log checks we used to rely on. Every cluster, every Oracle source, every job — visible right here. Let me walk you through how it works."

Before recording: clear your browser notifications, set Helios to full screen, and zoom to 110% so text is readable in the recording.
2
Protection Group setup key moment
~60 sec
What to show on screen
  • Navigate to Data Protection → Protection Groups
  • Click into an existing Oracle Protection Group (don't create a new one — faster to show a live configured one)
  • Highlight the Oracle source list — show which databases are included
  • Show the schedule tab — daily incremental, weekly baseline
  • Point out the Cohesity Agent connection status (green = healthy)
Script

"Here's our Oracle Protection Group. This replaced every custom RMAN bash script we had. The Cohesity Agent connects natively to Oracle — it supports standalone instances, RAC, and ASM — with zero custom scripting on our end. You can see the databases enrolled here. The schedule is set to incremental forever — meaning after the initial full baseline, we only ever move changed data blocks. That's what shrunk our backup window from 8 to 12 hours down to 30 to 45 minutes."

Move your mouse slowly and deliberately when hovering over UI elements. Audiences lose track of fast cursor movement on recordings.

Policy configuration & WORM immutability key moment
~60 sec
What to show on screen
  • Navigate to the Protection Policy attached to your Oracle group
  • Show the retention settings — highlight the WORM lock period field
  • Show the DataLock / SnapLock toggle — confirm it is enabled
  • If possible, try to delete or modify a locked snapshot — show that the action is blocked
  • Show the "Locked until" timestamp on a completed snapshot
Script

"This is where compliance is enforced. Our Protection Policy has DataLock enabled — that's Cohesity's WORM implementation. Once a snapshot is taken, it's locked for the retention period you see here. And I mean locked — watch what happens if I try to delete it. The platform refuses. Not even a Cohesity admin account can override this during the lock window. This is exactly what our previous NFS setup could not do — and it's what makes us audit-ready."

  • If you can't demo a live deletion attempt, just point to the lock icon on the snapshot and state: "This padlock icon means this snapshot is legally immutable until the date shown."
This is the most important 20 seconds of the entire video. Pause slightly before and after the WORM demo moment to let it land.
4
Helios monitoring dashboard
~75 sec
What to show on screen
  • Go to the Helios global dashboard — show the job success/failure timeline
  • Click into a completed Oracle backup job — show the run details: duration, data transferred, dedup ratio
  • Navigate to Alerts — show the alerting configuration (email / webhook / SNMP)
  • Show the Data Reduction section — highlight the global dedup ratio number
  • If visible, show the anomaly detection / security panel
Script

"Before Cohesity, verifying backup health meant SSH-ing into servers and grepping through log files. Now this is what we see every morning — a single view across every Oracle environment. Here's last night's run for our production database: job duration, data transferred, and the dedup ratio — you can see we're achieving over 3 to 1 data reduction globally, which directly reduces storage costs. Alerts fire immediately if anything deviates — email, webhook, SNMP — before a human would ever notice a problem. And this anomaly detection panel uses machine learning to flag unusual data patterns like sudden large-scale encryption, which is the signature of ransomware."

When reading the dedup ratio number on screen, pause on it for 2 seconds before speaking. Let the audience see it before you explain it.

Recovery workflow key moment
~45 sec
What to show on screen
  • Go to Data Protection → Recover → Oracle
  • Select a source database and show the snapshot calendar — multiple restore points visible
  • Show the recovery options: full DB restore, table-level, or clone
  • Initiate a test clone to a non-production target (ideal) — or just show the recovery wizard up to the confirmation screen without executing
  • If a recovery was previously done, navigate to the job history and show a completed recovery run time
Script

"Finally, recovery — which was the most painful part of our old setup. To restore from legacy backups, we'd manually locate the right backup pieces on the NFS share, figure out the correct RMAN sequence, and pull everything back over the network. It could take hours. With Cohesity, I select the database, pick a point-in-time snapshot from this calendar, choose where to recover to — and the data is already local and indexed. Recovery goes from hours to minutes. For dev and QA, we can even spin up a zero-storage clone of a multi-terabyte database instantly, without touching production."

End the recording here with a clean pause, then stop. Don't trail off — a confident stop is better than a long outro.
✓
Before you hit record — checklist
  • Set screen resolution to 1920×1080, browser zoom to 110%
  • Use a quiet room — mic quality matters more than camera
  • Do one full dry run from start to finish before recording
  • Have all Helios tabs pre-loaded and ready to switch to
  • Use a tool like OBS, Loom, or QuickTime for screen capture with voiceover
  • Speak 20% slower than feels natural — recordings always sound faster on playback
  • Keep the cursor visible and moving intentionally — don't let it drift
  • If you stumble, pause for 2 seconds and restart that sentence — easy to edit out
  • Aim for 4–5 min total — tight is better than thorough for a presentation insert
